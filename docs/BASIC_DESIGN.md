# ライブ記録アプリ 基本設計書

> 日本全国の「行ったライブ」を公演単位で記録し、ランキングや実績で自慢できる Web アプリ（のちにネイティブアプリ化）

---

## 1. 前提・方針

| 項目 | 方針 |
|------|------|
| **無料でできる範囲** | 開発・運用は可能な限り無料枠（Free tier）で構成する |
| **データの軸** | **公演中心** … 「〇〇のライブ（公演）にこの会場で行った」を1件として記録 |
| **リリース順** | まず **Web** で公開 → のちに **ネイティブアプリ**（React Native / Flutter 等）へ展開 |
| **会場データ** | **ハイブリッド方式**（後述「会場データの方針」） |

---

## 2. 会場データの方針【おすすめ】

**結論: ハイブリッド（主要会場を事前登録 ＋ ユーザー登録可）**

### 比較

| 方式 | メリット | デメリット |
|------|----------|------------|
| **ユーザー登録制のみ** | 運用コストほぼゼロ、どんな会場でも登録できる | 初速で検索が空になりがち、表記ゆれ・重複が多い |
| **事前にデータを入れるのみ** | 初日から検索しやすい、表記が統一される | 収集・更新の手間、マイナー会場は網羅しづらい |

### 推奨: ハイブリッド

1. **初期データ**  
   - 主要会場（Zepp、LIQUIDROOM、新宿LOFT、赤坂BLITZ など）を **数十〜100件程度** 手動または公開リストから投入。  
   - 無料でできる範囲: Wikipedia、公式サイト、まとめサイト等から名前・都道府県・市区町村程度を取得。
2. **ユーザー登録**  
   - 会場が一覧にない場合だけ「会場を登録」できるようにする。  
   - 登録時は「会場名・都道府県・市区町村（任意）」だけ必須にし、重複チェック（同じ名前＋都道府県）で弾く。
3. **運用**  
   - 最初は審査なしで即時反映でよい。のちに荒らしが増えたら「報告・非表示」や簡易モデレーションを検討。

これで「初日から使える」かつ「マイナー会場もユーザーが足せる」バランスが取れる。

## 3. 無料枠で使える技術（目安）

| 分野 | 候補 | 無料枠のイメージ |
|------|------|------------------|
| **ホスティング** | Vercel, Netlify | 個人用途なら無料 |
| **DB・BaaS** | Supabase, Firebase (Firestore + Auth) | 小規模なら無料枠内 |
| **認証** | Supabase Auth / Firebase Auth | 月あたり一定数まで無料 |
| **地図** | Leaflet + 国土地理院タイル / OpenStreetMap | タイル利用は無料 |
| **画像生成（シェアカード）** | クライアントで html2canvas 等 | サーバー不要で無料 |
| **アナリティクス** | Google Analytics 4, Plausible 等 | 無料枠あり |

※ 実際の利用量に応じて、Supabase/Firebase の無料枠を超えないよう設計する（読み書き回数・ストレージの見積もりを後で行う）。

## 4. コア機能（公演中心）

- **公演の記録**  
  - 1件 = 「〇〇（アーティスト/イベント名）のライブに、△△会場で、YYYY/MM/DD で行った」  
  - 会場は検索 or ユーザー登録で選択。
- **マイ一覧・マップ**  
  - 行った公演リスト、および日本地図上で「行った会場」をピン表示。
- **ランキング**  
  - 公演数、会場数、都道府県数など（後述）。
- **プロフィール・実績**  
  - 総公演数、会場数、都道府県数、バッジ。
- **シェア用カード**  
  - 実績やランキングを1枚画像で出力し、SNS に貼れるようにする。

---

## 5. データモデル（概要）

- **users** … 認証用 ID、表示名、アバター等。
- **venues** … 会場（名前、都道府県、市区町村、緯度経度任意）。事前投入＋ユーザー登録。
- **events / lives** … 公演（会場 ID、日付、アーティスト/イベント名、メモ等）。ユーザーが「行った」を登録するときに作成 or 選択。
- **attendances** … 参加記録（user_id, event_id または venue_id + 日付 + イベント名）。公演中心なら「event に user が1件紐付く」形。

公演を「会場＋日付＋名前」の組み合わせで一意にし、同じ公演に複数ユーザーが紐付ける形にすると、ランキングや「この公演に行った人」が作りやすい。

---

## 6. 画面構成（Web）

1. **ホーム** … 自分のサマリ、直近の記録、ランキングへの導線。
2. **記録する** … 会場を検索/登録 → 日付・公演名（アーティスト名）を入力して保存。
3. **マイ記録** … 一覧・マップ。
4. **ランキング** … 公演数 / 会場数 / 都道府県数など。
5. **プロフィール** … 実績・バッジ・シェア用カード作成。
